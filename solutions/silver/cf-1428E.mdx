<Spoiler title="Hint 1">

If you have a carrot and know the number of pieces you want to cut it into, how would you cut it to minimize the time to eat for all the segments of that carrot?

</Spoiler>

<Spoiler title="Hint 2">

Calculate the time to eat when a carrot is cut into $1$, $2$, $3$, and more times.
What do you notice about these times?

</Spoiler>

<Spoiler title="Hint 3">

In hint $2$, what do you observe about the *change* in those times as the numbee of cuts increase?

</Spoiler>

<Spoiler title="Solution">

## Explanation

The sum of squares of two numbers given a fixed sum is minimized when the two numbers are equal.
If we extend that to this problem under each carrot, then they should be cut into pieces of lengths that are as close together as possible, without any pair of lengths differing by more than $1$.
Specifically, if a carrot of length $l$ is cut into $t$ parts, there will be $t-l\bmod t$ of pieces with length $\lfloor\frac lt\rfloor$, and $l\bmod t$ pieces with length $\lfloor\frac lt\rfloor+1$.
Let the time taken to eat these pieces be $f(l,p)$, which can be calculated in $\mathcal O(1)$ time.

Now, the problem reduces to deciding the number of cuts on each carrot.
Inuititively, we guess that the more pieces we cut a carrot into, the more the total time gets smaller.
And if we manually work out the time for each number of cuts, we see that the decrease in total time gets smaller as well.

To approximately see why, assume that the cuts are perfectly even, allowing fractional lengths;
in that case, $f(l, t)\approx t(\frac lt)^2=\frac{l^2}t$.
Note that since the function is in the form of $g(x)=\frac ax$, $f$ decreases and does so slower as $t$ increases;
this means that when incrementing the number of cuts on a carrot, each incrementation will save time, but the saving will decrease as we go further.

This inspires a greedy solution;
we have set carrots to be cut into $1$ segment initially (uncut), and precompute the decrease in time of that carrot if the number of cuts on it is increased by $1$.
We maintain a priority queue of carrots sorted by the change described above;
for $k-n$ times, we take the carrot with the largest change value and increase its number of cuts by $1$.
We then recompute the time of eating that carrot, and the change to that time if the number of cuts is *further* increased by $1$ before reenqueing the carrot.

As the returns are diminishing for each increase in the cuts, there cannot be a situation where picking a suboptimal cut first will result in better cuts later for a carrot.

## Implementation

**Time Complexity:** $\mathcal{O}(k\log n)$

<CPPSection>

```cpp
#include <iostream>
#include <queue>
using namespace std;
// Returns the total time for consumption when a
// carrot of size l is split into t parts.
long long calculate_time(int l, int t) {
	// First, give each segment l / t; then, there's l % t
	// total left, so there's l % t of (l / t + 1) segments, and
	// the rest are l / t.
	return 1ll * (l % t) * (l / t + 1) * (l / t + 1) +
	       1ll * (t - l % t) * (l / t) * (l / t);
}
struct Carrot {
	int length;      // Original length of carrot
	int cut;         // Current consideration for amount of cuts
	long long cost;  // Total time to eat all cut segments
	long long diff;  // If cut is increased by 1, the change to cost
	// Update cost and diff based on length and cut.
	void update_cache() {
		this->cost = calculate_time(length, cut);
		this->diff = this->cost - calculate_time(length, cut + 1);
	}
};
bool operator<(const Carrot &a, const Carrot &b) { return a.diff < b.diff; }

int main() {
	int n, k;
	cin >> n >> k;
	long long ans = 0;
	priority_queue<Carrot> carrots;
	for (int i = 1; i <= n; i++) {
		Carrot carrot;
		cin >> carrot.length;
		carrot.cut = 1;
		carrot.update_cache();
		carrots.push(carrot);
		ans += carrot.cost;
	}
	for (int i = 1; i <= k - n; i++) {
		auto front = carrots.top();
		carrots.pop();
		ans -= front.diff;
		front.cut++;
		front.update_cache();
		carrots.push(front);
	}
	cout << ans << endl;
}
```
